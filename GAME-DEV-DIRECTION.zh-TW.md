# 遊戲企劃與開發方向（Agent 工作基準）

## 專案定位

本專案目標是開發一款類似 Soul Knight 的 2D Roguelike 動作遊戲。
玩家選擇角色後進入關卡迷宮探索，透過戰鬥與傳送陣推進節點，最終進入 Boss 房並擊敗 Boss 完成結算。

技術方向以 Phaser 3 為主要遊戲框架，資產來源以本專案輸出的 sprite 與 map 素材為主。

## 核心玩法循環

1. 選擇角色
2. 進入起始地圖節點
3. 清除當前地圖敵人
4. 尋找並啟動傳送祭壇
5. 移動至下一個地圖節點
6. 重複探索與戰鬥直到 Boss 節點
7. 擊敗 Boss 後進行通關結算

## 關卡與地圖結構

關卡不是單一路徑，而是由多張地圖節點透過傳送陣形成網狀迷宮。
每個節點可有不同屬性（戰鬥、補給、事件、Boss 等），並可依條件動態開啟後續路徑。

## 戰鬥與技能設計原則

現階段普通攻擊只先處理角色動作本體（action/attack）。
技能系統採用組合式設計：角色動作 + 特效 + 判定 + 數值。

技能建議拆分為以下資料欄位：

1. action_key（角色動作）
2. cast_frame（出手幀）
3. hit_shape（判定形狀）
4. damage_profile（傷害係數）
5. effect_keys（對應特效）
6. cooldown 與 mana_cost
7. on_hit 行為（擊退、硬直、DOT 等）

## 背包與掉落設計

除了角色隊伍配置外，遊戲內還有背包系統。

目前規劃：

1. 背包欄位上限為 4 格
2. 地圖中撿拾的道具可進入背包
3. 道具本身可以帶有詞綴 tag
4. 道具 tag 可計入羈絆條件，加速隊伍羈絆的湊齊

設計原則：

1. 背包系統先以少量欄位維持選擇壓力，不做過大容量
2. 道具優先支援被動效果與 tag 補足，不急著加入複雜主動施放道具
3. 道具 tag 應與角色 tag 使用同一套羈絆判定邏輯
4. 背包資料結構需可支援掉落、拾取、替換、丟棄與結算保存

## 目前資產物件分類（精簡版）

建議以 5 大類作為主資料夾與資料結構基準：

1. Unit：玩家、敵人、NPC、召喚單位
2. VFX：角色掛載特效、斬擊、命中特效、光效
3. Projectile：火球、箭矢、氣功等飛行物
4. Map：地圖節點、圖層、傳送連結、碰撞
5. Prop：場景物件（可穿透/不可穿透/可破壞/可互動）

## 地圖編輯器方向（新增）

專案將包含遊戲內或外掛式地圖編輯器，用於動態建立與調整迷宮節點連結。

編輯器目標：

1. 建立地圖節點（新增/刪除/命名）
2. 設定節點類型（戰鬥、補給、事件、Boss）
3. 設定節點之間的傳送連線（有向/無向）
4. 編輯每張地圖的物件配置與碰撞資訊
5. 支援即時預覽路徑連通性
6. 儲存為 JSON，供 Phaser 執行時載入

最小資料模型建議：

- map_nodes
- map_edges
- spawn_rules
- portal_rules
- prop_layout
- collision_layout

## 開發分期（Roadmap）

### Phase 0：核心基底

1. Phaser 專案骨架與場景切換
2. 基礎輸入（上下左右、普攻、技能）
3. 角色狀態機（idle/walk/attack/hurt/death）
4. 血量與魔力系統

### Phase 1：最小可玩版本（MVP）

1. 1 位玩家角色
2. 2 種一般敵人 + 1 位 Boss
3. 基本地圖節點推進（5-8 節點）
4. 傳送陣切圖流程
5. 可破壞物件與補給掉落

### Phase 2：內容擴充

1. 多角色與多敵人行為
2. 技能組合系統（action + VFX + projectile）
3. 地圖節點類型擴充（事件、補給、精英）
4. 關卡池與隨機生成規則
5. 背包系統與道具掉落
6. 道具詞綴 tag 與羈絆計數串接

### Phase 3：隊伍與羈絆

1. 每關結束三選一夥伴
2. 隊伍上限 3 名
3. 角色 tag 詞綴系統
4. 羈絆門檻與效果啟動

### Phase 4：地圖編輯器整合

1. 編輯器建立節點與連線
2. 編輯器保存/載入 JSON
3. 即時驗證迷宮可通關性
4. 與遊戲流程一鍵測試

### Phase 5：長期成長與局外養成

以下內容屬於後期擴充，可在核心玩法穩定後再實作：

1. 通關金幣結算
2. 經驗值結算
3. 金幣用於解鎖新角色
4. 金幣用於解鎖天賦系統或開局額外加成
5. 經驗值用於提高角色等級並強化能力
6. 長期養成資料保存與進度讀取

設計原則：

1. 金幣屬於局外資源，主要影響解鎖與永久成長
2. 經驗值屬於角色成長資源，可用於基礎能力提升
3. 天賦系統先以被動增益為主，避免過早影響核心平衡
4. 此階段不阻塞主遊戲循環，應在核心戰鬥與關卡流程成熟後再接入

## Agent 執行原則

後續所有 agent 任務請遵循以下優先順序：

1. 先完成可玩核心，再擴充系統深度
2. 優先資料驅動，不把規則硬寫死在場景程式
3. 角色動作與特效分離，避免資產耦合
4. 地圖採節點化設計，方便編輯器接入
5. 每新增一個系統，需可在最小測試場景中驗證

## 補充說明

背景劇情目前可留白，待核心玩法與系統循環穩定後再補。
現階段開發重心是可玩性、系統可擴充性、以及資產管線一致性。
